## Commands

```bash
npm run dev    # Start dev server on port 3000 (opens browser)
npm run build  # Production build → dist/
```

No test suite or linter is configured.

## Architecture

### Frontend (React SPA)

The app is a single-page React application with **custom History API routing** — there is no React Router. All routing logic lives in `src/App.tsx`:

- `parsePath()` maps `window.location.pathname` to a `ViewType`
- `pushPath()` / `go()` push URLs and re-parse state
- `popstate` listener handles browser back/forward

Views: `home | archives | categories | tags | about | article | tagged | category | admin`

Route components are **lazy-loaded** (`React.lazy`) so each page only loads its chunk on first visit. Heavy article dependencies (KaTeX, syntax highlighter) are **dynamically imported inside `Article.tsx`** only when the markdown content contains math (`$...$`) or fenced code blocks — articles without those pay zero cost.

### Data layer

Posts are fetched from a **Supabase Edge Function** (REST API) and cached in React state. On error or empty response, `src/data/fallbackPosts.ts` is used as the display source.

Two hooks handle data:
- `useBlogPosts` — fetches the list (content stripped server-side for performance)
- `useBlogPost(id)` — fetches a single post with full content; skipped when list already has content

### Backend (Supabase Edge Function)

`src/supabase/functions/server/index.tsx` — Hono router running on Deno, deployed as a Supabase Edge Function. All routes are prefixed `/make-server-860c354e/`.

Storage is a key-value abstraction over a Supabase Postgres table (`kv_store_860c354e`). Posts are stored as `post:{id}` keys where `id` is a Unix timestamp string.

**Do not edit these autogenerated files:**
- `src/supabase/functions/server/kv_store.tsx` — KV store adapter
- `src/utils/supabase/info.tsx` — project ID and anon key

### Admin panel

Only rendered in `import.meta.env.DEV`. Accessible via the `+` button (bottom-right in dev) or `/admin` route. Supports creating, editing, and deleting posts. The markdown import feature parses YAML frontmatter from `.md` files (see `posts/` directory for an example).

### Styling

The app mixes **Tailwind utility classes** with a custom editorial design system defined in `src/index.css`. The design system uses CSS custom properties (`--bg-surface`, `--text-tertiary`, `--border-default`, etc.) toggled by a `dark` class on `<html>`. Named semantic classes like `editorial`, `meta`, `hero`, and `accent` are defined in the CSS and should be preferred over raw Tailwind for text/layout elements that need to match the editorial tone.

### Build output

Vite splits bundles into: `react-vendor`, `markdown-vendor`, and per-route chunks. All assets ≥10KB are emitted with `.gz` and `.br` companions. The `_headers` file in `public/` and `dist/` configures Cloudflare to serve pre-compressed files. `console.log` calls are stripped in production builds.
